# 问题定义与上位原则

## 1. 背景

本司已确定“物理 AI 大模型”为既定战略，因此训练所需的高质量、高多样性交互数据是**刚性需求**。
战略核心不是“要不要投入数据”，而是**用什么模式以最低综合成本、最快速度获得足够多样且足够有价值的数据**。

## 2. 上位机制：数据飞轮（长期战略，不是短期门槛）

数据基础设施的长期形态是一个**双环耦合飞轮**：

```
内环（能力建设，当前阶段重心）     外环（能力外化，长期目标）
自采数据  →  模型能力  →  对外部署 / 生态开放
   ↑                              ↓
   └─  回流 ego / 本体 / 视频 / 长尾  ──┘
             （多样性与规模）
```

- **内环**：自有场景自采 → 训练 → 模型能力。解决"从 0 到 1"。**这是 Stage 1 / Stage 2 的主战场。**
- **外环**：对外部署与生态开放 → 回流跨场景 / 跨本体 / ego 数据 → 扩大多样性。解决"从 1 到 N"。**这是 Stage 3 才真正转起来的引擎。**

### 关于飞轮的关键判断（避免误用叙事）

飞轮能转有两个技术前置条件当下都未满足：

1. 底座模型能稳定消化异构来源数据（跨场景 / 跨本体 / 跨视角）；
2. 外部部署的规模和数据权属治理能支撑回流。

在这两个条件满足前，**外环不是"主引擎"，只是"保留选项"**。过早把飞轮当主路径，会演变为"用长期叙事绑架短期执行"。

因此本战略的真实打法是：

- **当下（Stage 1 / Stage 2）**：全力打磨内环，外环只做 schema 与接口的**便宜保留**（不落地）。
- **中期（Stage 2 末 / Stage 3 初）**：底座技术与数据权属成熟后，才启动外环真实开放。
- **长期（Stage 3 及以后）**：飞轮成为第二增长曲线。

## 3. 两类问题（主线不变）

### Problem A：数据底座建设（内环，主线）

- 目标：在自有场景完成从采集到模型迭代的周级闭环，单位有效数据成本持续下降。
- 关键杠杆：自采 / 外采 / 治理自动化 / 增量训练。
- 衡量指标：`单位有效数据成本`、`数据采纳率`、`质检通过率`、`UDV（价值密度）`。
- **当前阶段资源分配 >= 80%。**

### Problem B：飞轮外化与生态开放（外环，长期战略，当前只做保留）

- 当前阶段目标：冻结接口 / 数据权属 / 评测规范 schema，**不做真实对外开放**。
- 中长期目标：当 A 达标且跨场景 UDV 可验证后，启动真实外化，获取多样性回流。
- 衡量指标（未来启用）：`回流多样性`、`跨场景 UDV`、`回流有效小时占比`。
- **当前阶段资源分配 <= 10%（仅 schema 与规范类工作）。**

### Problem C：模型与本体商业化（下游，不在本文档范畴）

- 机器人本体销售、服务、场景运营等业务层变现不在本战略域内讨论。

## 4. 决策原则

1. **A 是主线，B 是保留**。A 的每周级稳定度是任何 B 动作的前提。
2. **飞轮是长期战略叙事**，不作为 Stage 2 的硬门槛。Stage 2 只负责让"飞轮可以启动"，不负责让"飞轮转起来"。
3. 任何加速 B 的提议必须先回答：
   - 是否延误 A 的周级闭环？
   - 底座模型是否已证明能利用异构回流数据？
4. 任何新采集/接入模式必须先回答：是否降低 A 的单位成本，或为未来 B 的启动降低切换成本。
5. 生态开放（2nd-dev / 商业伙伴）的节奏：Stage 1 末冻结 schema → Stage 2 仅对 1-2 家战略伙伴试点 → Stage 3 真正规模化。

## 5. 文档分工

- `00-problem-definition.md`：本文件，上位原则与飞轮机制。
- `01-data-cost-model.md`：数据来源、成本模型、飞轮增益、生态外化路径。
- `02-stage-milestones.md`：阶段门槛，含飞轮与生态开放门槛。
- `03-data-infra-v1.md`：技术执行蓝图（含外部回流接口预留）。
- `archive/`：历史与可选策略资料。

## 6. 上位判断（给决策者看的四句话）

- 数据是确定性战略投入，不是商业赌注。
- 当下（12-18 个月内）核心 ROI = 单位数据成本下降 × 模型价值密度。
- 飞轮是 24-36 个月的长期引擎，不是 Stage 2 的执行门槛；Stage 2 只需把"飞轮可以启动"的接口和规范准备好。
- 飞轮真正转起来的前提是：底座模型能消化异构数据。这个技术条件没达到前，所有外环加速都是空中楼阁。

# 数据来源、成本与飞轮模型（核心决策文档）

> 上位原则见 `00-problem-definition.md`。本文件回答三个问题：
> 1) 如何以最低综合成本获得足够高质量的训练数据（内环 / Problem A）
> 2) 如何通过外部部署与生态开放放大数据多样性（外环 / Problem B 的数据价值）
> 3) 外部变现作为次要副产物如何处理（外环的商业价值）

## 1. 决策目标（主目标 + 长期保留目标）

本阶段的决策目标是**单主目标 + 一个保留目标**，不是双目标并行：

- **主目标（Stage 1 / Stage 2 全力投入）**：`MinimizeTotalCost` = Sum_i (Hours_i × UnitCost_i) / ValueDensity_i
- **保留目标（Stage 3 才真正启用）**：`MaximizeFlywheelGain` = Diversity(外部回流) × Scale(外部回流) × 跨场景 UDV

当下阶段 ROI 口径：

`ROI_now = (成本下降节约 + 模型能力价值折算) / 总投入`

当飞轮启动前置条件满足后（见 §4.2），才把飞轮增益并入 ROI 公式。**过早并入会导致战略被长期叙事绑架。**

## 2. UDV：价值密度指标

`UDV = ΔSuccessRate / EffectiveDataHours_added`

- 内环用途：比较自采/外采/治理路径的价值密度。
- 外环用途：度量回流数据是否真的扩展了模型能力边界（`跨场景 UDV`）。
- 有效小时计入条件：完成质检 + 带任务语义标签 + 被训练消费 + 对应模型版本已部署。

## 3. 数据来源五路径对比（TCO + 飞轮框架）

相比纯成本比较，加入"飞轮贡献"列。

| 模式 | 单位成本 | 价值密度 | 多样性贡献 | 起量速度 | 质量可控 | 合规风险 | 战略定位 |
|---|---|---|---|---|---|---|---|
| 自采（真机+自有场景） | 高 | 高 | 低 | 慢 | 高 | 低 | **Stage 1/2 主力** |
| 外采（第三方数据服务商） | 中 | 中 | 中 | 快 | 中 | 中 | **Stage 1/2 补盲** |
| 生态（云/硬件伙伴共建） | 低-中 | 中-高 | 中-高 | 中 | 中 | 中 | Stage 2 末期试点 |
| 众包（消费级设备+C端） | 低 | 低 | 高 | 慢-中 | 低 | 高 | Stage 3 潜力，前期不投 |
| 2nd-dev 生态开放 | 低（边际成本） | 视伙伴场景 | 高（跨行业/跨本体） | 中 | 需架构保障 | 中 | **Stage 3 主引擎，Stage 1/2 仅留接口** |

### 选型原则

- 起步期（Stage 1）：自采 + 外采为主；同时冻结对外回流 SDK 与数据权属 schema（便宜保险，不落地）。
- 爬坡期（Stage 2）：仍以自采 + 外采为主；可对 1-2 家战略生态伙伴做受控试点，但不作为主要数据来源。
- 规模期（Stage 3）：2nd-dev 生态与众包才成为多样性主力；此时飞轮真正转起来。

**反模式提示**：把"2nd-dev 生态开放"当作 Stage 2 的数据来源主力，是一种常见的战略漂移——看似激进进取，实际是把未经验证的长期赌注当成短期确定性来源，底座团队会被外部接口维护与合作方支持稀释。

## 4. 数据飞轮模型（长期形态，非当前执行项）

### 4.1 飞轮的四个节点

1. **自采** → 自有场景数据 → 训练底座模型
2. **外化** → 模型能力部署到客户 / 伙伴 / 2nd-dev 开发者
3. **回流** → 客户与伙伴场景的新数据（ego 视频、跨本体、跨行业长尾）回到治理层
4. **再训练** → 模型能力边界扩展，下一轮外化更广

### 4.2 飞轮启动的真实前置条件（全部满足才启动）

1. **技术条件**：底座模型在跨场景 / 跨本体 / 跨视角迁移上有可量化的能力证据（非当前学术 SOTA 的水平）。
2. **工程条件**：内环周级闭环稳定 ≥ 12 周（见 `03-data-infra-v1.md` 第 7 节）。
3. **接口条件**：外部回流 SDK 与数据权属 schema 已冻结并通过内外审计。
4. **合规条件**：数据脱敏、跨境、伙伴授权条款完整。

**当前（2026 年）这 4 个条件中，条件 1 是最硬的卡点，也是行业共性难题。**

### 4.3 飞轮失速预警（启动后监控）

- 外部回流数据的跨场景 UDV 持续为负 → 模型无法利用新多样性 → 回流只是成本负担
- 回流数据采纳率 < 50% → 接入架构或数据质量有结构性问题

### 4.4 为什么不能提前启动

- 底座不稳时开放接口 → 接口维护成本稀释内环资源；
- 底座平庸时开放能力 → 把自身能力短板暴露给合作方和竞品；
- 数据治理未自动化时接入外部 → 治理成本可能高于自采；
- 伙伴场景分布未知时承诺 SLA → 产品口碑风险。

结论：**飞轮的魅力是长期，代价是短期**。Stage 1/2 只做"便宜保留"，不做"真实启动"。

## 5. 单位成本拆解与降本路径

`UnitCost = (设备摊销 + 人力 + 场地 + 治理 + 算力分摊 + 合规) / 有效小时`

降本优先级（按 ROI 排序）：

1. 治理自动化（质检/对齐/切片）——短期 ROI 最高
2. 采集标准化工具链——中期降边际成本
3. 多源融合（自采+外采+生态）——中期结构性降本
4. 飞轮外环放量——**长期潜力**，边际成本趋零，但要求底座模型能利用异构数据
5. 众包试点（条件受控）——长期潜力但高不确定性

## 6. 假设清单

| ID | 假设 | 当前值 | 目标值 | 证据来源 |
|---|---|---|---|---|
| H1 | 自采单位成本 | 待填 | 按季度下降 >=10% | 内部场景运营 |
| H2 | 外采单位成本 | 待填 | 低于自采 30% | 外部报价样本 |
| H3 | 众包单位成本 | 待填 | 低于自采 50% 且 UDV > 阈值 | 受控试点 |
| H4 | 生态共建成本 | 待填 | 通过共建降摊销 | 合作协议 |
| H5 | 治理自动化率 | 待填 | >=70% | 工具链指标 |
| H6 | 2nd-dev 生态回流多样性 | 待填 | 外部场景数 >= 自有 2 倍 | 生态接入日志 |
| H7 | 跨场景 UDV | 待填 | 不低于自有场景 UDV 的 70% | 模型评测 |

## 7. 外环路径一：对外付费部署（Stage 2 窗口）

- 定位：向同场景行业客户交付"模型 + 数据闭环"解决方案。
- 当前首要产出：行业标杆 + 回流通道验证（不是规模化营收，也不是主数据源）。
- 时机：内环达 Stage 1→2 所有硬门槛后启动（见 `02-stage-milestones.md` §3）。
- 规模上限：Stage 2 期外部部署数量 ≤ 2 家，避免底座团队被稀释。
- 详细执行参考：`archive/external-pilot-playbook.md`（激活时再启用）。

## 8. 外环路径二：2nd-dev 生态开放（Stage 3 主引擎）

### 8.1 当前阶段（Stage 1 / Stage 2）只做什么

**只做便宜保留，不做真实开放**：

- Stage 1 末：冻结接口规范与数据权属 schema（见 `03-data-infra-v1.md` §6）
- Stage 2 全程：仅对 1-2 家战略伙伴做**非公开的受控联合验证**，不对任何外部开发者开放

不做什么：

- 不做公开 API 文档、不做开发者门户、不做 SDK 公开发布
- 不承接 2nd-dev 伙伴的技术支持、不投入 DevRel 人力
- 不以"生态伙伴数"作为任何团队的 KPI

### 8.2 Stage 3 启动条件（必须全部满足）

1. 内环跨场景 UDV 连续 8 周为正
2. 底座模型能可量化地利用异构回流数据（跨本体 UDV ≥ 自有场景 UDV × 0.7）
3. DataOps / Compiler / Eval 完成标准化版本
4. 至少 2 家 Stage 2 伙伴的受控验证产出正向回流证据

### 8.3 Stage 3 开放对象（分层）

按接入门槛从低到高，**但进入顺序由 Stage 3 启动后再决策**：

- 模型能力 API / 技能调用接口（黑盒能力）
- 数据接入 SDK（允许伙伴把场景数据回流到平台）
- 评测与失败模式库（让伙伴自助诊断）
- 精调与 LoRA 分发（让伙伴在平台上训练特化能力）

### 8.4 开放边界（避免战略漂移）

- 不对外开放底座训练数据原始权属
- 不承诺跨行业泛化与重度定制效果
- 外部收入占总成本上限：Stage 2 ≤ 10%，Stage 3 ≤ 50%

### 8.5 为什么不能在 Stage 2 提前开放（技术判断）

- 学界 SOTA 的跨本体/跨视角迁移效果仍不稳定，强行开放等于把能力短板公开；
- 平台生态网络效应依赖"核心能力显著领先"，咱们现在不是这样；
- API 和 SDK 的向后兼容一旦承诺，重构成本极高，不宜过早冻结。

## 9. 外环路径三：工具链 / 标准授权（长线）

- DataOps / Compiler / Eval 标准化后对行业授权。
- 仅在 Stage 3 开启，不占用 Stage 1-2 资源。
- 价值偏行业位势，直接营收不作为主指标。

## 10. 与下游文档的衔接

- 阶段门槛（`02-stage-milestones.md`）：飞轮作为 Stage 3 门槛，Stage 2 仅要求"接口就绪度"。
- 执行蓝图（`03-data-infra-v1.md`）：五层架构中为外部回流预留接口与权属 schema（规范冻结，不落地）。
- 对外试点（`archive/external-pilot-playbook.md`）：仅在 Stage 2 门槛满足后激活，规模受控。

# 阶段里程碑与技术成熟度门槛

> 上位原则见 `00-problem-definition.md`；数据来源与飞轮模型见 `01-data-cost-model.md`。
> 本文件回答：**什么时候可以进入下一阶段？** 核心判据是**底座技术成熟度 + 飞轮启动条件**，不是流程或组织。

## 1. 阶段划分

- `Stage 1`：内环底座建设（0-6 个月）。自采闭环 + 外部接入规范**冻结但不落地**。
- `Stage 2`：底座稳定 + 有限外化（6-18 个月）。1-2 家战略客户受控试点；2nd-dev 生态保持仅接口保留，不公开开放。
- `Stage 3`：飞轮真正启动 + 平台化（18-30 个月）。多源回流成为多样性主力。

阶段切换由三类判据触发：**底座技术成熟度、数据成本下降、有限外化验证**。组织流程不作为切换条件。

**关键原则**：飞轮真正转起来是 Stage 3 的事。Stage 2 的任务是**让飞轮可启动**，不是**让飞轮转起来**——不要把 Stage 3 的指标前置到 Stage 2。

## 2. 门槛优先级（固定不可颠倒）

`底座成本门槛 > 底座技术门槛 > 数据可用性门槛 > 接口就绪门槛 > 产品门槛 > 商业门槛`

上位门槛不满足，下位达标也不得跨阶段。
注：此处不再把"飞轮门槛"作为 Stage 2 的主门槛——飞轮成熟门槛移至 Stage 2 → Stage 3 的准入位置。

## 3. Stage 1 → Stage 2 准入门槛

### 3.1 底座成本门槛（必须满足）

- 自采单位成本连续 2 个季度下降 >= 10%（基准=季度首月）
- 治理自动化率 >= 60%
- 有效采纳率 >= 70%（入池数据被训练消费）

### 3.2 底座技术门槛（必须满足）

- `UDV_total > 0` 且连续 6 周不反转
- 至少 3 类核心任务中，2 类 `UDV_by_task > 0`
- 连续 2 次版本迭代无 P0 回归

### 3.3 数据可用性门槛（必须满足）

- 连续 2 个月有效数据小时稳定增长
- 质检通过率 >= 92%
- 失败样本占比保持目标区间（20%-35%）

### 3.4 接口就绪门槛（必须满足，轻量）

仅要求**规范冻结**，不要求真实开放：

- 外部回流 SDK 规范 V1 冻结并通过内部 Mock 测试
- 数据权属 schema V1 冻结且法务审核通过
- MVP 能力边界文档冻结（可开放 / 不可开放能力清单）

### 3.5 产品化准备门槛（满足 2/3 即可）

- 试点标准验收协议冻结
- 版本发布节奏可预测（过去 6 周按节奏发布 >= 80%）
- 至少 1 套评测规范文档化可对外讲清

## 4. Stage 2 运行目标（非硬门槛，仅过程目标）

Stage 2 是"底座稳定 + 有限外化"的窗口，**不是飞轮启动窗口**。本阶段的运行目标不作为跨阶段硬门槛，仅作为过程验证：

### 4.1 底座持续迭代（主线）

- 内环周级闭环连续运行 >= 20 周不中断
- UDV_total 稳定提升，至少 2 类核心任务 `UDV_by_task` 每季度改善
- 自采单位成本持续下降（每季度 >= 5%）

### 4.2 有限外化试点（副线，规模受控）

- 最多 2 家同场景行业客户付费试点，目标是验证"可交付性"，不是规模营收
- 最多 1-2 家战略伙伴非公开联合验证，验证接口规范可用性
- 外部部署总规模不超过内部门店规模的 30%

### 4.3 Stage 3 门槛前置证据收集（副线）

- 积累跨场景迁移数据，为 Stage 3 跨场景 UDV 评估做准备
- 不设"回流有效小时占比"硬指标——该指标移至 Stage 3 准入门槛

### 4.4 明确不做的事（避免战略漂移）

以下动作在 Stage 2 不启动：

- 2nd-dev 生态公开开放、开发者门户、公开 SDK 发布
- "生态伙伴数"作为 KPI
- 外部回流数据混入主训练集（只走隔离池）
- 承诺跨行业泛化或重度定制能力

## 5. Stage 2 → Stage 3 准入门槛

### 5.1 飞轮启动门槛（必须满足，**本阶段最硬门槛**）

- 底座模型跨场景 / 跨本体 UDV 连续 8 周为正（证明模型可消化异构数据）
- 外部回流有效小时占比 >= 15% 且连续 8 周稳定（从 Stage 2 末期的小规模回流启动）
- 跨场景 UDV 与自有场景 UDV 差值 <= 30%
- Stage 2 期试点客户/伙伴中，至少 2 家产出正向回流证据

注：**若模型无法证明能利用异构数据（第 1 条），即使其他门槛满足也不启动 Stage 3**。这是整个战略最硬的技术卡点。

### 5.2 技术产品门槛（必须满足）

- 跨场景（门店/行业）迁移性能损耗 <= 20%
- 新客户 / 新伙伴上线周期 <= 10 周
- 模型版本回归自动化覆盖 >= 80%

### 5.3 平台能力门槛（满足 2/3 即可）

- DataOps / Compiler / Eval 三模块完成标准化版本
- 数据接口规范或评测规范至少发布 1 套并稳定使用
- 失败模式知识库覆盖核心任务 Top20 异常

### 5.4 商业验证门槛（满足 2/3 即可，不作为主触发）

- 外部 PoC 转付费率 >= 40%
- 外部收入占总成本 >= 10% 且连续 1 季度为正毛利
- 至少 1 套标准接口被合作方规模化接入

### 5.5 Stage 3 储备模块

- 机器人技能数据生态平台（消费级头戴采集 + 技能封装分发）
  - 评估记录：`archive/bp-assessment-skill-ecosystem.md`
  - 采纳方案：降级为 Stage 3 可选模块
  - 重评触发：VLA 第一视角视频到真机策略在受控任务达 >= 60%

## 6. 加速通道（仅窗口期启用）

允许在以下情形提前局部外化：

- 战略客户绑定硬件大单且内部节奏可控
- 监管或生态合作窗口期短

启用条件：

- 3.1 / 3.2 / 3.3 / 3.4 所有硬门槛全部达标
- 仅开放 MVP 能力，不开放定制
- 明确加速期限与回退阈值

## 7. 决策输出（每月一次，产品总监视角）

每评估周期只输出三个结论：

1. `是否跨阶段`：通过 / 延期 / 有条件通过
2. `当前阶段内是否加码外化`：是 / 否 / 维持
3. `补哪些短板`：数据、模型、工程、成本、接口就绪

## 8. 预警与回退（技术触发）

出现以下任一情况，触发回退评估：

- 底座成本门槛连续 2 个季度不达标
- Stage 2 期外部试点占用资源导致内部里程碑延误 > 20%
- Stage 3 启动后跨场景 UDV 连续 2 个迭代反转（证明模型无法利用异构数据）

回退动作：

- 暂停新增外部试点与接入
- 资源优先回补内环闭环
- 4 周内完成修复并复评

# Data Infra v1 执行蓝图（技术视角）

> 上位原则见 `00-problem-definition.md`，数据来源与飞轮模型见 `01-data-cost-model.md`，阶段门槛见 `02-stage-milestones.md`。
> 本文件只讲技术执行：**架构、节奏、数据对象、外部接入预留、验收**。不涉及组织与流程。

## 1. 目标

0-12 个月打通"采集 → 治理 → 编译 → 训练 → 部署 → 回流"的周级闭环，输出每周可验证的模型迭代。
**同时**为 Stage 2 的外部回流和 2nd-dev 生态接入预留接口与 schema（规范冻结，不落地）。

## 2. 首批场景

零售/服务高频任务（自有场景，用于成本控制与价值密度验证）：

- 补货（shelf_restock）
- 取放（pick_place）
- 分拣（light_sorting）
- 简单导购交互（guide_interaction）

## 3. 五层架构（内外环统一设计）

| 层 | 内环职责（v1 落地） | 外环预留（v1 出规范，Stage 2 落地） |
|---|---|---|
| 采集（Collection） | 自采 + 遥操作 + 故障复现 | 外部回流接入 SDK 规范 |
| 治理（Governance） | 多模态对齐、质检、标签补全 | 外部数据脱敏与合规校验流水线规范 |
| 编译（Compiler） | 切片、标签体系、检索索引 | 跨来源（自采/外采/生态/2nd-dev）溯源字段 |
| 训练（Training） | 复用自驾训练栈，周级窗口 | 来源加权 & 采样策略接口（先留钩子） |
| 评估/部署（Eval/Deploy） | 离线评测 + 门店灰度 + 回流 | 跨场景 UDV 评测协议、失败模式回传协议 |

核心原则：**v1 只做内环，但所有数据对象和接口都按"未来要接外环"的 schema 设计**。避免 Stage 2 返工。

## 4. 周级闭环节拍

- 周一：基于上周失败模式定义本周采集配额
- 周二到周三：采集 + 治理，输出可训练数据包
- 周四：触发训练流水线并完成离线评估
- 周五：1-2 家门店灰度部署，线上表现回流至治理层

## 5. 数据对象与最小字段（含飞轮溯源字段）

### 5.1 采集记录 `collection_record`

- `record_id` / `scene_id` / `task_type` / `robot_version`
- `operator_mode`（teleop/human_assist/autonomous）
- `start_ts` / `end_ts`
- **新增**：`source_type`（self_collect / external_purchase / ecosystem / second_dev_partner / crowd）
- **新增**：`source_id`（伙伴/客户/众包用户匿名 ID）
- **新增**：`data_rights`（own / licensed / shared / restricted）
- **新增**：`view_type`（first_person_ego / third_person / mixed）

### 5.2 编译样本 `compiled_sample`

- `sample_id` / `source_record_id`
- `task_phase` / `outcome`（success/failure/recovery）/ `failure_mode`
- `quality_score` / `data_version`
- **新增**：`provenance_chain`（来源链路，含脱敏处理记录）
- **新增**：`usage_policy`（训练 / 评测 / 内部共享 / 外部共享）

### 5.3 部署回流 `deployment_feedback`

- `model_version` / `scene_id` / `task_type`
- `success_flag` / `human_intervention_flag` / `incident_level`
- **新增**：`deployment_channel`（internal / external_customer / second_dev_partner）
- **新增**：`cross_scene_flag`（是否跨出自有场景覆盖范围）

## 6. 外部接入规范（v1 冻结，Stage 2 激活）

### 6.1 外部回流 SDK 规范（接口冻结）

- 数据接入协议：鉴权、批量上传、增量同步、断点续传
- 元数据必填字段：`source_type` / `source_id` / `view_type` / `task_type_hint`
- 默认脱敏策略：人脸 / 语音 / 位置 / 文本敏感信息的自动处理

### 6.2 数据权属 schema（Stage 1 末冻结）

- 所有权 / 使用权 / 衍生模型权分离
- 合作伙伴四种授权档：`only_self_use` / `share_anonymized` / `share_full` / `platform_full`
- 合同条款与 schema 字段一一对应，避免合同-数据脱节

### 6.3 2nd-dev 开放接口（只出文档）

- 模型能力 API：技能列表、调用参数、限速与计费骨架
- 数据接入 SDK：复用 6.1 的接入协议
- 评测接入 API：提交评测集并获取跨场景 UDV 报告

规范冻结不等于落地。v1 阶段只产出文档和 Mock 实现，真实对外服务在 Stage 2 早期开启。

## 7. v1 技术验收门槛（12 个月内）

### 7.1 闭环效率

- `LoopCycleTime <= 7 天`（采集到灰度上线）
- 周级迭代连续运行 >= 8 周不中断

### 7.2 数据有效性

- 质检通过率 >= 92%
- 数据采纳率 >= 70%（入池被训练消费）
- 失败样本占比在目标区间（20%-35%）

### 7.3 模型价值

- 核心任务加权成功率较基线提升 >= 15%
- 人工介入率较基线下降 >= 20%
- `UDV_total > 0`，且至少 2 类任务 `UDV_by_task > 0`

### 7.4 可复制性

- 至少 3 个门店复用同一流程
- 任务模板复用率 >= 60%
- 跨门店迁移性能损耗 <= 20%

### 7.5 飞轮接入就绪度（新增）

- 外部回流 SDK 规范 V1 冻结并通过内部 Mock 测试
- 数据权属 schema V1 冻结且法务审核通过
- 2nd-dev 接口文档 V1 可独立被外部研发跑通 Demo

## 8. 12 个月里程碑（按季度）

- Q1：五层架构 MVP + 单门店闭环跑通
- Q2：扩展多门店，统一标签与失败模式库；启动外部接入 schema 设计
- Q3：周级发布稳定，自动化指标与回归报表；外部 SDK 与权属 schema 内部评审
- Q4：输出"内部闭环能力包"（能力边界、接口、评测协议）+ "外部接入规范 V1"，作为 Stage 2 飞轮启动的技术底座

## 9. 关键技术风险与缓解

- 风险：采集量增长但高价值样本不足。
  - 缓解：失败样本占比下限强制约束；每周复盘失败模式覆盖。
- 风险：模型无法有效利用新增数据（成本升、价值不升）。
  - 缓解：跟踪数据采纳率、训练贡献度、UDV；连续 2 周反转触发训练策略复盘。
- 风险：跨门店迁移损耗超限。
  - 缓解：场景差异向量化标注；迁移时强制最小增量微调样本量。
- 风险：灰度事故回滚不及时。
  - 缓解：回滚演练每季度 1 次；失败门店白名单控制。
- 风险（飞轮）：外部接入 schema 设计不足，Stage 2 被迫返工重构数据底座。
  - 缓解：Q2 启动 schema 设计，Q3 完成内部评审，Q4 冻结并 Mock 跑通。
- 风险（飞轮）：外部回流数据质量差，污染内环训练。
  - 缓解：外部数据默认进隔离池，通过独立评测通道转正，不直接混入主训练集。
